# Настройки проекта

В этой папке лежат настройки проекта **OZON Similar Products**.

Настройки вынесены в отдельные YAML-файлы, чтобы запускать один и тот же код с разными параметрами: для быстрой
локальной проверки, полноценного построения рекомендаций, оценки качества и подбора параметров.

## Какие файлы здесь лежат

| Файл                                        | За что отвечает                                                          |
|---------------------------------------------|--------------------------------------------------------------------------|
| `paths.yaml`                                | пути к данным, результатам, исходному коду и обязательным папкам проекта |
| `data.yaml`                                 | ожидаемые архивы, названия колонок и известные типы действий             |
| `baseline.yaml`                             | базовые параметры для обычного запуска и проверки идеи                   |
| `production.yaml`                           | более строгие параметры для основного построения рекомендаций            |
| `evaluation.yaml`                           | настройки оценки качества рекомендаций                                   |
| `tuning/search_space.yaml`                  | общее пространство подбора параметров                                    |
| `tuning/search_space_scoring_core.yaml`     | подбор параметров оценки похожести без резервных рекомендаций            |
| `tuning/search_space_scoring_fallback.yaml` | подбор параметров оценки похожести вместе с резервными рекомендациями    |

## Какой файл использовать

### Для первого запуска

Используйте `configs/baseline.yaml`.

Он подходит для проверки, что проект запускается, данные читаются, промежуточные таблицы создаются, а рекомендации
сохраняются в ожидаемом формате.

Пример:

```bash
uv run python scripts/run_pipeline.py 2024-04-23 --lookback-days 7 --top-k 20 --config-path configs/baseline.yaml
```

### Для основного запуска

Используйте `configs/production.yaml`.

В нём находятся параметры, которые ближе к основному сценарию построения рекомендаций: больше итоговых рекомендаций,
строже фильтры для пар товаров и включены резервные рекомендации.

Пример:

```bash
uv run python scripts/run_full.py 2024-04-23 --lookback-days 1 --validation-days 1 --top-k 20 --config-path configs/production.yaml
```

### Для оценки качества

Используйте `configs/evaluation.yaml`.

Этот файл описывает, какие действия считать релевантными, какие веса им назначать и какие метрики считать.

### Для подбора параметров

Используйте файлы из `configs/tuning/`.

Пример:

```bash
uv run python scripts/run_tune.py 2024-04-23 --lookback-days 1 --validation-days 1 --top-k 20 --config-path configs/production.yaml --search-space-path configs/tuning/search_space.yaml --max-trials 30 --tuning-strategy random
```

## `paths.yaml`

Файл `paths.yaml` задаёт основные пути проекта.

В нём описаны:

* где лежат исходные архивы;
* где должны появиться подготовленные данные;
* куда сохраняются результаты запусков;
* какие папки исходного кода считаются обязательными;
* какие служебные директории должны существовать в проекте.

Этот файл используется не для настройки качества рекомендаций, а для проверки структуры проекта и согласованности путей.

Обычно его не нужно менять.

Менять `paths.yaml` стоит только если меняется структура репозитория.

## `data.yaml`

Файл `data.yaml` описывает входные данные.

В нём задаются:

* названия ожидаемых архивов;
* правила поиска parquet-файлов внутри распакованных данных;
* обязательные колонки таблицы товаров;
* обязательные колонки таблицы пользовательских событий;
* известные типы действий пользователей.

Главные сущности:

```text
product_information
user_actions
raw_data
```

`product_information` описывает данные о товарах.

`user_actions` описывает пользовательские события.

`raw_data` содержит служебные настройки для подготовленных данных.

Этот файл нужно менять осторожно: если изменить названия колонок или ожидаемые типы действий, нужно убедиться, что код
чтения данных, валидация и дальнейшие этапы обработки поддерживают эти изменения.

## `baseline.yaml`

Файл `baseline.yaml` содержит базовые параметры построения рекомендаций.

Он подходит для разработки, локальной проверки и первых экспериментов.

В нём задаются:

* размер временного окна;
* время разрыва между пользовательскими сессиями;
* максимальное количество товаров в одной сессии;
* количество рекомендаций для каждого товара;
* список товарных действий;
* приоритет действий внутри сессии;
* параметры графа товаров;
* параметры оценки похожести;
* пути к промежуточным артефактам;
* настройки резервных рекомендаций;
* базовые настройки оценки качества.

Этот файл можно использовать как безопасную стартовую точку для экспериментов.

## `production.yaml`

Файл `production.yaml` содержит параметры основного запуска.

Он похож на `baseline.yaml`, но рассчитан на более строгий сценарий построения рекомендаций.

В нём, например, могут отличаться:

* время разрыва между сессиями;
* минимальные требования к парам товаров;
* веса действий при расчёте похожести;
* количество итоговых рекомендаций;
* параметры резервных рекомендаций;
* правила включения товаров с малым числом событий.

Если нужно улучшать качество рекомендаций, чаще всего изменения будут происходить именно здесь.

## Основные группы настроек

### `pipeline`

Настройки полного запуска.

Здесь задаются:

* `session_timeout_minutes` — через сколько минут бездействия начинать новую сессию;
* `max_items_per_session` — сколько товаров максимум брать из одной сессии;
* `top_k` — сколько рекомендаций строить для каждого товара;
* `lookback_days` — сколько дней брать в расчёт;
* `update_strategy` — стратегия обновления;
* размеры служебных пакетов и корзин для обработки данных.

### `events`

Настройки пользовательских действий.

Здесь задаётся, какие действия считаются товарными:

```text
view
click
favorite
to_cart
```

Также отдельно указывается действие `search`, потому что поиск не является товарным действием, но может встречаться в
исходных событиях.

### `item_pair_builder`

Настройки построения пар товаров внутри сессии.

Главный параметр здесь — `signal_priority`.

Он задаёт, какое действие считать более сильным, если один и тот же товар встретился в сессии с разными типами действий.

Например:

```yaml
signal_priority:
  view: 1
  click: 2
  favorite: 3
  to_cart: 4
```

Важно: этот приоритет не является итоговым весом рекомендации. Он нужен только для выбора самого сильного сигнала внутри
одной сессии.

### `graph`

Настройки графа товаров.

Здесь находятся параметры ослабления связи между товарами:

* по расстоянию внутри сессии;
* по давности события.

В текущей базовой схеме эти механизмы можно отключать или включать через `enabled`.

Эти настройки стоит менять осторожно, потому что они влияют на то, какие пары товаров считаются сильными до финального
расчёта похожести.

### `scoring`

Настройки итоговой оценки похожести.

Здесь задаются:

* метод расчёта;
* источник счётчиков;
* преобразование счётчиков;
* веса действий;
* минимальные требования к парам товаров;
* калибровка;
* нормализация по популярности товара.

Чаще всего при улучшении качества меняются именно эти параметры:

```text
scoring.business_weights
scoring.beta
scoring.min_pair_count
scoring.min_unique_users
scoring.min_unique_sessions
scoring.normalize_by_item_popularity
scoring.popularity_normalization
```

### `artifacts`

Пути к промежуточным данным.

Здесь указано, куда сохраняются:

* очищенные события;
* состояние сессий;
* популярность товаров;
* распределение действий;
* дневные пары товаров;
* агрегированные пары товаров.

Обычно эти пути не нужно менять.

### `outputs`

Пути к итоговым результатам.

Главные директории:

```text
outputs/
outputs/latest/
```

`outputs/latest/` хранит последнюю опубликованную версию рекомендаций.

### `evaluation`

Настройки оценки качества.

Здесь задаются:

* режим релевантности;
* веса действий для оценки;
* действия, которые считаются значимыми для ранжирования;
* минимальный порог релевантности.

Например, добавление в корзину обычно считается более сильным сигналом, чем просмотр.

### `business.fallback`

Настройки резервных рекомендаций.

Резервные рекомендации нужны для товаров, которым не хватило поведенческих данных.

Здесь задаются:

* включён ли резервный слой;
* сколько кандидатов брать;
* какие уровни резервных рекомендаций использовать;
* учитывать ли бренд;
* использовать ли глобально популярные товары;
* включать ли товары с малым числом событий.

## `evaluation.yaml`

Файл `evaluation.yaml` отдельно описывает оценку качества.

В нём задаются:

* веса действий;
* список действий, важных для ранжирования;
* тип разбиения данных;
* список метрик;
* значения `k`, для которых считаются метрики.

Основные метрики:

```text
recall_at_k
hitrate_at_k
ndcg_at_k
coverage
diversity
```

Этот файл полезен, когда нужно менять не сам алгоритм рекомендаций, а способ проверки качества.

## `tuning/`

Папка `tuning/` содержит пространства подбора параметров.

Каждый файл описывает:

* целевую метрику;
* вспомогательные метрики;
* штрафные метрики;
* ограничения;
* параметры, которые можно перебрать;
* возможные значения или диапазоны значений.

### `search_space.yaml`

Общее пространство подбора параметров.

Подходит для широкого эксперимента, где одновременно подбираются параметры оценки похожести, количества рекомендаций и
резервного слоя.

### `search_space_scoring_core.yaml`

Более узкий подбор параметров оценки похожести.

Резервные рекомендации здесь отключены, поэтому такой запуск помогает понять качество основной поведенческой логики без
дополнительного заполнения недостающих рекомендаций.

### `search_space_scoring_fallback.yaml`

Подбор параметров оценки похожести вместе с резервными рекомендациями.

Этот файл полезен, когда нужно настроить не только основную выдачу, но и поведение для товаров с малым количеством
событий.

## Что можно менять безопасно

| Что менять                                     | Где                                                       |
|------------------------------------------------|-----------------------------------------------------------|
| Количество рекомендаций                        | `topk.top_k`, `pipeline.top_k`, `business.fallback.top_k` |
| Размер временного окна                         | `pipeline.lookback_days`                                  |
| Время разрыва между сессиями                   | `pipeline.session_timeout_minutes`                        |
| Веса действий                                  | `scoring.business_weights`                                |
| Минимальное число встреч пары                  | `scoring.min_pair_count`                                  |
| Минимальное число сессий                       | `scoring.min_unique_sessions`                             |
| Включение нормализации по популярности         | `scoring.normalize_by_item_popularity`                    |
| Включение резервных рекомендаций               | `business.fallback.enabled`                               |
| Использование бренда в резервных рекомендациях | `business.fallback.enable_brand`                          |
| Метрики качества                               | `evaluation.metrics`                                      |
| Пространство подбора параметров                | `configs/tuning/*.yaml`                                   |

## Что менять осторожно

| Что менять                          | Почему осторожно                                          |
|-------------------------------------|-----------------------------------------------------------|
| `data.expected_columns`             | Эти колонки проверяются при чтении данных                 |
| `data.known_action_types`           | От этого зависит обработка пользовательских действий      |
| `product_information.id_column`     | Этот идентификатор используется по всему проекту          |
| `artifacts.*_dir`                   | Эти пути связывают этапы конвейера обработки              |
| `outputs.latest_dir`                | На эту папку ориентируется просмотр последнего результата |
| `item_pair_builder.signal_priority` | Это влияет на выбор самого сильного сигнала внутри сессии |
| `graph.distance_decay`              | Это меняет силу связей между товарами внутри сессии       |
| `graph.time_decay`                  | Это меняет вклад старых и новых событий                   |
| `evaluation.relevance_weights`      | Это меняет смысл оценки качества                          |

## Что не стоит хранить в настройках

В файлах из `configs/` не должно быть:

* паролей;
* токенов;
* личных путей конкретного пользователя;
* больших списков товаров;
* временных результатов экспериментов;
* локальных абсолютных путей, если можно использовать относительные.

Настройки должны оставаться воспроизводимыми для другого человека, который откроет репозиторий.

## Типовые сценарии

### Проверить, что проект запускается

```bash
uv run python scripts/run_pipeline.py 2024-04-23 --lookback-days 7 --top-k 20 --config-path configs/baseline.yaml
```

### Запустить основной сценарий

```bash
uv run python scripts/run_full.py 2024-04-23 --lookback-days 1 --validation-days 1 --top-k 20 --config-path configs/production.yaml
```

### Подобрать параметры

```bash
uv run python scripts/run_tune.py 2024-04-23 --lookback-days 1 --validation-days 1 --top-k 20 --config-path configs/production.yaml --search-space-path configs/tuning/search_space.yaml --max-trials 30 --tuning-strategy random
```

### Посмотреть последний результат

```bash
uv run python scripts/preview_latest_recommendations.py
```

## Коротко

```text
paths.yaml       → где что лежит
data.yaml        → какие данные ожидаются
baseline.yaml    → базовый запуск
production.yaml  → основной запуск
evaluation.yaml  → оценка качества
tuning/          → подбор параметров
```

Если нужно быстро проверить проект, используйте `baseline.yaml`.

Если нужно строить основной результат, используйте `production.yaml`.

Если нужно улучшать качество, меняйте параметры в `production.yaml` и проверяйте их через `evaluation.yaml` или файлы из
`tuning/`.
