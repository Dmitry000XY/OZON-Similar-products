# Построение похожих товаров

В этом модуле строится основная поведенческая часть рекомендаций.

На вход приходят пользовательские сессии и статистики по товарам. На выходе получается ранжированный список похожих
товаров для каждого `item_id`.

Идея простая: если товары часто встречаются в одних пользовательских сессиях, между ними есть поведенческая связь. Чем
чаще и сильнее эта связь, тем выше товар поднимается в списке похожих.

## Что делает модуль

Модуль `retrieval` отвечает за четыре этапа:

```text
сессии
→ пары товаров
→ агрегированные пары
→ оценка похожести
→ top-K кандидатов
```

В текущей реализации здесь находятся:

| Класс                 | За что отвечает                                                    |
|-----------------------|--------------------------------------------------------------------|
| `ItemPairBuilder`     | строит направленные пары товаров внутри пользовательских сессий    |
| `PairAggregator`      | объединяет дневные пары за выбранное временное окно                |
| `CoVisitationScorer`  | считает итоговую оценку похожести                                  |
| `TopKSelector`        | выбирает лучшие похожие товары для каждого исходного товара        |
| `DistanceDecayConfig` | настраивает вклад пары с учётом расстояния между товарами в сессии |
| `TimeDecayConfig`     | настраивает вклад пары с учётом давности события                   |

## Основные файлы

| Файл                                       | Что в нём находится                                    |
|--------------------------------------------|--------------------------------------------------------|
| [`build_pairs.py`](build_pairs.py)         | построение пар товаров из сессий                       |
| [`aggregate_pairs.py`](aggregate_pairs.py) | агрегация дневных пар за временное окно                |
| [`scoring.py`](scoring.py)                 | расчёт оценки похожести                                |
| [`topk.py`](topk.py)                       | выбор лучших кандидатов для каждого товара             |
| [`decay.py`](decay.py)                     | настройки дополнительных весов по расстоянию и времени |

## Построение пар товаров

### `ItemPairBuilder`

`ItemPairBuilder` берёт пользовательские сессии и строит направленные пары товаров.

Если в одной сессии пользователь взаимодействовал с товарами A, B и C, то из этой сессии можно получить пары:

```text
A → B
A → C
B → A
B → C
C → A
C → B
```

Пары направленные: связь `A → B` и связь `B → A` хранятся отдельно.

Это важно, потому что в дальнейшем мы строим рекомендации вида:

```text
для item_id найти похожие similar_item_id
```

Пример использования:

```python
from ozon_similar_products.retrieval.build_pairs import ItemPairBuilder

builder = ItemPairBuilder(max_items_per_session=50)

daily_pairs = builder.transform_day(sessions)
```

## Как выбирается сигнал товара внутри сессии

Один и тот же товар может встретиться в сессии несколько раз и с разными действиями.

Например:

```text
view
click
to_cart
```

Мы не хотим создавать лишние дубли одного и того же товара внутри сессии. Поэтому `ItemPairBuilder` схлопывает повторные
действия товара до одного самого сильного сигнала.

Приоритет задаётся в настройках:

```yaml
item_pair_builder:
  signal_priority:
    view: 1
    click: 2
    favorite: 3
    to_cart: 4
```

Важно: этот приоритет не является итоговым весом рекомендации. Он нужен только для выбора самого сильного действия
товара внутри одной сессии.

Итоговые веса действий применяются позже, в `CoVisitationScorer`.

## Какие сессии используются для пар

Мы не строим пары из сессий, в которых слишком мало или слишком много товаров.

Сессия должна содержать хотя бы два разных товара. Слишком длинные сессии ограничиваются параметром:

```yaml
pipeline:
  max_items_per_session: 50
```

Это нужно, чтобы не создавать шумные связи из очень больших сессий, где товары могли оказаться рядом случайно.

Подробнее о сессиях: [`../preprocessing/README.md`](../preprocessing/README.md).

## Дневные статистики пар

Кроме сырых строк пар, `ItemPairBuilder` умеет строить компактные дневные статистики через `build_daily_pair_stats`.

На выходе получается объект `DailyPairStats`:

```text
counts
user_keys
session_keys
raw_pair_rows
```

Что хранится внутри:

| Поле            | Что означает                           |
|-----------------|----------------------------------------|
| `counts`        | счётчики пар по действиям              |
| `user_keys`     | уникальные пользователи для пары       |
| `session_keys`  | уникальные сессии для пары             |
| `raw_pair_rows` | сколько сырых строк пар было построено |

Такой формат нужен, чтобы не хранить и не сканировать слишком много сырых строк при последующей агрегации.

Контракты дневных статистик описаны в [`../../../docs/data_contract.md`](../../../docs/data_contract.md).

## Дополнительные веса связей

В модуле есть два необязательных механизма, которые меняют вклад пары до финального расчёта `score`.

Первый — вес по расстоянию внутри сессии. Он нужен, если мы хотим считать товары, расположенные ближе друг к другу в
сессии, более связанными.

Второй — вес по давности. Он нужен, если мы хотим сильнее учитывать свежие события и слабее учитывать старые.

Оба механизма задаются в блоке `graph`:

```yaml
graph:
  distance_decay:
    enabled: false
    strategy: none

  time_decay:
    enabled: false
    strategy: none
```

Поддерживаемые стратегии:

```text
none
weight_table
exponential
```

Если механизмы выключены, все пары получают обычный вес `1.0`.

Если они включены, результат сохраняется в `weighted_*`-счётчиках. Обычные фактические счётчики при этом тоже
сохраняются.

## Агрегация пар за окно

### `PairAggregator`

После построения дневных пар мы объединяем их за выбранный период.

Например, если запуск использует окно в 7 дней, агрегатор собирает пары за эти 7 дней и считает общие статистики.

```python
from ozon_similar_products.retrieval.aggregate_pairs import PairAggregator

aggregator = PairAggregator()

pair_aggregates = aggregator.aggregate_window(
    daily_pairs=[pairs_day_1, pairs_day_2],
    window_start="2024-04-01",
    window_end="2024-04-07",
)
```

На выходе получается таблица:

```text
item_id
similar_item_id
pair_count
view_count
click_count
favorite_count
to_cart_count
weighted_pair_count
weighted_view_count
weighted_click_count
weighted_favorite_count
weighted_to_cart_count
unique_users
unique_sessions
window_start
window_end
```

Эта таблица ещё не является рекомендациями. Это факты о том, как часто товары встречались вместе и какими действиями
была подкреплена связь.

## Почему веса не применяются слишком рано

В этом модуле есть важное разделение.

`ItemPairBuilder` и `PairAggregator` сохраняют факты:

```text
сколько раз пара встретилась
сколько было view
сколько было click
сколько было favorite
сколько было to_cart
сколько уникальных пользователей
сколько уникальных сессий
```

Они не решают, что `to_cart` в 8 раз важнее `view`.

Это делает `CoVisitationScorer`.

Так мы можем менять веса и калибровку без пересборки всех пар товаров.

## Расчёт оценки похожести

### `CoVisitationScorer`

`CoVisitationScorer` получает агрегированные пары и считает итоговый `score`.

Пример:

```python
from ozon_similar_products.retrieval.scoring import CoVisitationScorer

scorer = CoVisitationScorer.from_config(config)

pair_scores = scorer.score(
    pair_aggregates=pair_aggregates,
    item_popularity=item_popularity,
)
```

Поддерживаются два метода:

```text
pair_count
calibrated_multichannel
```

`pair_count` — простой базовый вариант: чем чаще товары встречались вместе, тем выше оценка.

`calibrated_multichannel` — основной вариант: он учитывает отдельные каналы действий и их веса.

## Основной метод оценки

В методе `calibrated_multichannel` мы считаем оценку по отдельным типам действий:

```text
view_count
click_count
favorite_count
to_cart_count
→ score
```

Для каждого действия задаётся бизнес-вес:

```yaml
scoring:
  business_weights:
    view: 1.0
    click: 3.0
    favorite: 6.0
    to_cart: 8.0
```

Дальше счётчики могут быть преобразованы:

```yaml
scoring:
  count_transform:
    method: log
    smoothing: 1.0
```

Поддерживаемые преобразования:

```text
linear
sqrt
log
```

Это нужно, чтобы очень частые пары не забивали все остальные только за счёт большого количества повторов.

## Калибровка по частоте действий

Разные действия встречаются с разной частотой.

Просмотров обычно намного больше, чем добавлений в корзину. Поэтому редкие, но сильные действия можно дополнительно
усилить через калибровку.

За это отвечают параметры:

```yaml
scoring:
  beta: 0.5
  reference_action_type: view
  max_frequency_boost:
    view: 1.0
    click: 10.0
    favorite: 15.0
    to_cart: 30.0
```

Идея такая:

* `business_weights` задают смысловую важность действия;
* статистика частоты действий показывает, насколько действие редкое;
* `beta` регулирует силу частотной калибровки;
* `max_frequency_boost` не даёт редким действиям получить слишком большой множитель.

Распределение действий считается в [`features`](../features/README.md).

## Сырые и взвешенные счётчики

Scorer может использовать два источника счётчиков:

```text
raw
weighted
```

`raw` — обычные фактические счётчики.

`weighted` — счётчики после дополнительных поправок из блока `graph`.

Параметр задаётся так:

```yaml
scoring:
  count_source: raw
```

Если используется `weighted`, в агрегированной таблице должны быть `weighted_*`-колонки.

## Фильтры качества перед оценкой

Перед расчётом или после расчёта можно отсечь слабые пары.

Основные параметры:

```yaml
scoring:
  min_pair_count: 1
  min_unique_users: 1
  min_unique_sessions: 1
  min_weighted_pair_count: null
  min_score: null
```

Эти фильтры помогают убрать случайные связи между товарами.

Например, если пара встретилась один раз у одного пользователя, она может быть менее надёжной, чем пара, которая
встретилась у нескольких пользователей и в нескольких сессиях.

## Нормализация по популярности

Очень популярные товары могут часто встречаться почти со всем каталогом.

Чтобы такие товары не перетягивали выдачу на себя, scorer умеет делить оценку на популярность исходного товара и
кандидата.

Параметры:

```yaml
scoring:
  normalize_by_item_popularity: false
  popularity_normalization:
    popularity_column: unique_users
    smoothing: 1.0
    power: 0.5
```

Если `normalize_by_item_popularity` включён, в `score` нужно передать `item_popularity`.

Популярность товаров считается в [`features`](../features/README.md).

## Выбор top-K

### `TopKSelector`

После расчёта оценок у нас есть таблица пар с `score`.

`TopKSelector` превращает её в рекомендации:

```python
from ozon_similar_products.retrieval.topk import TopKSelector

selector = TopKSelector(top_k=20)

recommendations = selector.select(pair_scores)
```

Он делает несколько вещей:

* убирает строки без `item_id`;
* убирает строки без `similar_item_id`;
* убирает строки без `score`;
* не допускает рекомендацию товара самому себе;
* при необходимости применяет дополнительные пороги;
* убирает дубли пар;
* сортирует кандидатов;
* присваивает `rank`;
* оставляет только `top_k` товаров для каждого `item_id`.

На выходе получается таблица:

```text
item_id
similar_item_id
score
rank
source
```

Дополнительные диагностические колонки могут сохраняться вместе с рекомендациями, если они были во входной таблице.

## Откуда берётся `source`

Для рекомендаций, построенных по поведению пользователей, `TopKSelector` ставит:

```text
source = behavioral
```

Позже модуль [`business`](../business/README.md) может добавить резервные рекомендации с другими значениями `source`.

Так в итоговой таблице можно отличить обычные поведенческие рекомендации от резервных.

## Полный путь внутри модуля

```text
sessions
→ ItemPairBuilder
→ daily pair stats
→ PairAggregator
→ pair aggregates
→ CoVisitationScorer
→ pair scores
→ TopKSelector
→ behavioral recommendations
```

После этого рекомендации передаются в [`business`](../business/README.md), где при необходимости добавляются резервные
кандидаты, а затем в [`output`](../output/README.md), где результат сохраняется.

## Границы ответственности

Модуль `retrieval` отвечает за поведенческую выдачу похожих товаров.

Что делает `retrieval`:

* строит пары товаров внутри сессий;
* хранит счётчики по типам действий;
* считает уникальных пользователей и сессии для пары;
* агрегирует пары за временное окно;
* считает оценку похожести;
* выбирает top-K кандидатов.

Что не делает `retrieval`:

* не читает сырые данные из архивов;
* не очищает события;
* не строит пользовательские сессии;
* не считает популярность товаров с нуля;
* не добавляет резервные рекомендации;
* не сохраняет итоговые файлы;
* не читает готовые рекомендации для виджета.

Эти задачи находятся в других слоях:

| Задача                         | Модуль                                        |
|--------------------------------|-----------------------------------------------|
| чтение данных                  | [`data`](../data/README.md)                   |
| очистка событий и сессии       | [`preprocessing`](../preprocessing/README.md) |
| популярность товаров           | [`features`](../features/README.md)           |
| резервные рекомендации         | [`business`](../business/README.md)           |
| сохранение результата          | [`output`](../output/README.md)               |
| получение готовых рекомендаций | [`serving`](../serving/README.md)             |

## Что менять осторожно

| Что менять                     | Почему осторожно                                 |
|--------------------------------|--------------------------------------------------|
| `max_items_per_session`        | влияет на число пар и шум в длинных сессиях      |
| `signal_priority`              | меняет самый сильный сигнал товара внутри сессии |
| `graph.distance_decay`         | меняет вклад пар внутри одной сессии             |
| `graph.time_decay`             | меняет вклад пар из разных дней окна             |
| `business_weights`             | напрямую влияет на оценку похожести              |
| `beta`                         | меняет силу калибровки редких действий           |
| `count_transform`              | меняет влияние частых пар                        |
| `min_pair_count`               | отсекает слабые связи                            |
| `min_unique_users`             | влияет на устойчивость рекомендаций              |
| `min_unique_sessions`          | влияет на устойчивость рекомендаций              |
| `normalize_by_item_popularity` | может сильно изменить выдачу популярных товаров  |
| `top_k`                        | меняет размер итогового списка рекомендаций      |

После изменения этих параметров лучше запускать `ozon-run-full` и смотреть метрики качества.

## Быстрая проверка

Пример ручной проверки на готовых сессиях:

```python
from ozon_similar_products.retrieval.build_pairs import ItemPairBuilder
from ozon_similar_products.retrieval.aggregate_pairs import PairAggregator
from ozon_similar_products.retrieval.scoring import CoVisitationScorer
from ozon_similar_products.retrieval.topk import TopKSelector

pair_builder = ItemPairBuilder(max_items_per_session=50)
daily_pairs = pair_builder.transform_day(sessions)

aggregator = PairAggregator()
pair_aggregates = aggregator.aggregate_window(
    daily_pairs=[daily_pairs],
    window_start="2024-04-01",
    window_end="2024-04-01",
)

scorer = CoVisitationScorer(method="calibrated_multichannel")
pair_scores = scorer.score(pair_aggregates)

selector = TopKSelector(top_k=20)
recommendations = selector.select(pair_scores)

print(recommendations.head())
```

Для полного запуска лучше использовать команды из [`../../../scripts/README.md`](../../../scripts/README.md).

## Коротко

Мы используем `retrieval`, чтобы построить поведенческое ядро рекомендаций.

Сначала модуль строит пары товаров из пользовательских сессий.

Потом объединяет пары за временное окно.

После этого считает оценку похожести и выбирает лучшие товары для каждого `item_id`.

Резервные рекомендации, сохранение результата и выдача готового списка происходят уже в следующих слоях.
