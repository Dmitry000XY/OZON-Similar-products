# Признаки и статистики

В этом модуле мы считаем статистики по очищенным событиям.

После `preprocessing` у нас уже есть товарные действия пользователей: просмотры, клики, добавления в избранное и
добавления в корзину. На этом этапе мы не строим похожие товары напрямую, а собираем факты, которые потом используются в
расчёте похожести, резервных рекомендациях, оценке качества и диагностике.

Главный класс модуля — `ItemPopularityBuilder`.

## Что делает модуль

Модуль `features` отвечает за статистики по товарам и действиям.

```text id="ad66j4"
очищенные события
→ популярность товаров
→ распределение типов действий
→ диагностические срезы
```

Мы считаем:

* сколько раз товар встречался в событиях;
* сколько уникальных пользователей взаимодействовали с товаром;
* сколько было просмотров, кликов, добавлений в избранное и добавлений в корзину;
* как часто встречается каждый тип действия;
* какие товары популярны по дням, типам действий и виджетам.

## Основной файл

```text id="7rr9jw"
src/ozon_similar_products/features/
  item_popularity.py
```

В нём находится класс `ItemPopularityBuilder`.

Он работает с уже очищенными событиями и возвращает несколько таблиц:

| Метод                                  | Что возвращает                                               |
|----------------------------------------|--------------------------------------------------------------|
| `build_item_popularity`                | основную таблицу популярности товаров                        |
| `build_action_type_calibration_stats`  | распределение типов действий для калибровки оценки похожести |
| `build_item_popularity_by_date`        | диагностический срез популярности по датам                   |
| `build_item_popularity_by_action_type` | диагностический срез по типам действий                       |
| `build_item_popularity_by_widget_name` | диагностический срез по виджетам                             |

## Почему это отдельный слой

Мы специально отделяем расчёт статистик от расчёта рекомендаций.

На этом этапе мы сохраняем факты:

```text id="qagv6h"
товар 123 получил 100 просмотров
товар 123 добавили в корзину 7 раз
с действием to_cart связано 3% событий
```

Но мы пока не решаем, насколько один товар похож на другой.

Итоговая оценка похожести считается позже, в модуле `retrieval`.

Так проще контролировать проект: если рекомендации изменились, можно отдельно проверить, изменились ли сами данные,
популярность товаров, распределение действий или уже финальный расчёт похожести.

## Популярность товаров

### `build_item_popularity`

Этот метод строит основную таблицу популярности товаров.

Пример:

```python id="bw6el6"
from ozon_similar_products.features.item_popularity import ItemPopularityBuilder

builder = ItemPopularityBuilder()

item_popularity = builder.build_item_popularity(events_clean)
```

На вход подаются очищенные события.

На выходе получается таблица:

| Поле              | Что означает                   |
|-------------------|--------------------------------|
| `item_id`         | товар                          |
| `events_count`    | общее число товарных действий  |
| `unique_users`    | число уникальных пользователей |
| `views_count`     | число просмотров               |
| `clicks_count`    | число кликов                   |
| `favorites_count` | число добавлений в избранное   |
| `to_cart_count`   | число добавлений в корзину     |

Эта таблица используется дальше как фактическая статистика по товарам. Она полезна для резервных рекомендаций,
диагностики и возможной нормализации по популярности.

## Распределение типов действий

### `build_action_type_calibration_stats`

Этот метод считает, как часто встречается каждый тип действия на выбранном периоде.

Пример:

```python id="74vzsc"
action_distribution = builder.build_action_type_calibration_stats(
    events_clean=events_clean,
    calibration_start="2024-04-01",
    calibration_end="2024-04-07",
)
```

На выходе получается таблица:

```text id="rmekez"
action_type
events_count
event_share
unique_users
unique_items
calibration_start
calibration_end
```

Эта таблица нужна для калибровки оценки похожести.

Например, просмотров обычно намного больше, чем добавлений в корзину. Если не учитывать это различие, массовые просмотры
могут перекрыть более редкие, но более сильные действия. Поэтому мы сохраняем частоту действий отдельно, а уже потом
используем её в расчёте оценки похожести.

Важно: этот метод не назначает бизнес-веса действиям. Он только считает факты по выбранному периоду.

## Диагностические срезы

Помимо основных таблиц, в модуле есть дополнительные методы для анализа данных.

### По датам

```python id="10im5w"
popularity_by_date = builder.build_item_popularity_by_date(events_clean)
```

Этот срез помогает понять, как популярность товаров менялась по дням.

### По типам действий

```python id="w5psb4"
popularity_by_action = builder.build_item_popularity_by_action_type(events_clean)
```

Этот срез помогает увидеть, какие товары чаще просматривают, кликают, добавляют в избранное или корзину.

### По виджетам

```python id="e5f3xg"
popularity_by_widget = builder.build_item_popularity_by_widget_name(events_clean)
```

Этот срез полезен, если нужно понять, из каких мест интерфейса приходят взаимодействия с товарами.

Эти таблицы не обязательно нужны для каждого рабочего запуска, но они помогают проверять данные и объяснять поведение
рекомендаций.

## Какие действия учитываются

В модуле учитываются прямые товарные действия:

```text id="n48yl4"
view
click
favorite
to_cart
```

Они задаются в `ItemPopularityBuilder` как стандартный набор действий.

Если нужно использовать другой список действий, его можно передать явно:

```python id="70pp5t"
builder = ItemPopularityBuilder(
    item_action_types=["view", "click", "to_cart"],
)
```

Но менять этот список нужно осторожно: от него зависят популярность товаров, распределение действий и дальнейшая
калибровка.

## Границы ответственности

Модуль `features` не строит пары товаров и не рассчитывает итоговую похожесть.

Он отвечает за статистики:

```text id="x0vttu"
clean_events
→ item_popularity
→ action_type_distribution
```

Что делает `features`:

* считает популярность товаров;
* считает число уникальных пользователей по товарам;
* считает количество действий по типам;
* считает распределение типов действий;
* строит диагностические срезы.

Что не делает `features`:

* не строит пользовательские сессии;
* не создаёт пары товаров;
* не назначает итоговый вес похожести;
* не выбирает лучшие рекомендации;
* не добавляет резервные рекомендации;
* не сохраняет итоговый результат.

Эти задачи находятся в других слоях:

| Задача                          | Модуль          |
|---------------------------------|-----------------|
| очистка событий и сессии        | `preprocessing` |
| пары товаров и расчёт похожести | `retrieval`     |
| резервные рекомендации          | `business`      |
| оценка качества                 | `evaluation`    |
| сохранение результата           | `output`        |

## Как этот слой используется дальше

Основная таблица популярности используется в нескольких местах.

```text id="9xyh1a"
item_popularity
  → резервные рекомендации
  → диагностика популярности
  → возможная нормализация по популярности
```

Распределение типов действий используется для калибровки оценки похожести:

```text id="v3nmf3"
action_type_distribution
  → CoVisitationScorer
  → score похожести
```

То есть этот слой не принимает решение о рекомендациях, но даёт важные статистики для следующих этапов.

## Почему мы не агрегируем популярность из дневных таблиц

`ItemPopularityBuilder` считает популярность на нужном периоде по очищенным событиям.

Это сделано намеренно: для точного `unique_users` нельзя просто сложить дневные значения. Один и тот же пользователь мог
взаимодействовать с товаром в разные дни, и простая сумма переоценила бы число уникальных пользователей.

Поэтому, когда нужна точная популярность за окно, мы считаем её по событиям за всё окно.

## Быстрая проверка

Пример ручной проверки:

```python id="n9c1zu"
from ozon_similar_products.data import load_events
from ozon_similar_products.preprocessing.clean_events import EventCleaner
from ozon_similar_products.features.item_popularity import ItemPopularityBuilder

raw_events = load_events(sample_days=1, sample_rows=1000)

cleaner = EventCleaner(
    item_action_types=["view", "click", "favorite", "to_cart"],
)
events_clean = cleaner.transform_day(raw_events)

builder = ItemPopularityBuilder()

item_popularity = builder.build_item_popularity(events_clean)
action_distribution = builder.build_action_type_calibration_stats(
    events_clean=events_clean,
    calibration_start="2024-04-01",
    calibration_end="2024-04-01",
)

print(item_popularity.head())
print(action_distribution)
```

## Что менять осторожно

| Что менять                 | Почему осторожно                                             |
|----------------------------|--------------------------------------------------------------|
| список действий            | от него зависит, какие события попадут в статистики          |
| названия колонок счётчиков | эти поля используются следующими слоями                      |
| расчёт `unique_users`      | он влияет на популярность и диагностику                      |
| расчёт `event_share`       | он влияет на калибровку оценки похожести                     |
| сортировку результатов     | она может влиять на воспроизводимость диагностических таблиц |

Если меняется основная таблица популярности, нужно проверить модуль `business`, потому что резервные рекомендации
опираются на популярные товары.

Если меняется распределение типов действий, нужно проверить модуль `retrieval`, потому что оценка похожести может
использовать эти статистики для калибровки.

## Коротко

Мы используем `features`, чтобы посчитать фактические статистики по товарам и действиям.

Этот модуль не решает, какие товары похожи друг на друга.

Он готовит данные, которые дальше помогают считать похожесть, строить резервные рекомендации и проверять качество
результата.
