# Бизнес-правила рекомендаций

В этом модуле находятся правила, которые применяются после основной поведенческой выдачи.

Сейчас здесь реализован резервный слой рекомендаций. Он нужен для случаев, когда поведенческих данных не хватило: товар
новый, товар редко встречался в событиях или после фильтров осталось меньше `top_k` похожих товаров.

Основная логика похожести находится в [`retrieval`](../retrieval/README.md). Модуль `business` не пересчитывает `score`,
а дополняет уже готовые рекомендации понятными резервными кандидатами.

## Что делает модуль

```text
behavioral recommendations
+ item_popularity
+ product_information
→ recommendations with fallback
```

Модуль отвечает за три задачи:

1. Понять, включён ли резервный слой и какие правила использовать.
2. Построить списки резервных кандидатов по популярности и метаданным товаров.
3. Добавить недостающие рекомендации там, где поведенческой выдачи не хватило.

## Основной файл

```text
src/ozon_similar_products/business/
  fallback.py
```

В [`fallback.py`](fallback.py) находятся:

| Объект                     | За что отвечает                                    |
|----------------------------|----------------------------------------------------|
| `FallbackConfig`           | настройки резервного слоя                          |
| `FallbackCandidateBuilder` | глобальный список популярных товаров               |
| `FallbackIndexBuilder`     | индексы кандидатов по категории, типу и бренду     |
| `FallbackIndex`            | подготовленные списки резервных кандидатов         |
| `FallbackMerger`           | объединение поведенческих и резервных рекомендаций |
| `FallbackLayer`            | общий вход в резервный слой                        |

## Когда нужен резервный слой

Поведенческие рекомендации строятся по совместным действиям пользователей. Это хорошо работает для товаров, у которых
уже есть достаточно событий.

Но бывают случаи, когда данных мало:

* товар новый;
* товар редко просматривали;
* товар не попадал в сессии вместе с другими товарами;
* пары были отфильтрованы как слабые;
* после `TopKSelector` осталось меньше нужного числа рекомендаций.

В таких случаях мы не хотим оставлять пустой или слишком короткий список. Поэтому добавляем резервные рекомендации.

## Откуда берутся резервные кандидаты

Резервные кандидаты строятся из двух источников:

```text
item_popularity
product_information
```

`item_popularity` даёт понимание, какие товары в целом популярны.

`product_information` даёт метаданные товара: категорию, тип и бренд.

Сначала товары сортируются по популярности. Приоритет сортировки такой:

```text
unique_users
to_cart_count
favorites_count
clicks_count
views_count
events_count
item_id
```

После этого из популярных товаров строятся несколько уровней резервных кандидатов.

Подробнее:

* [`../features/README.md`](../features/README.md) — как считается `item_popularity`;
* [`../data/README.md`](../data/README.md) — как читается `product_information`.

## Уровни резервных рекомендаций

Резервный слой пробует кандидатов по порядку: сначала более близкие по метаданным, потом более общие.

```text
category + type
category
type
brand
global popular
```

В итоговой таблице источник рекомендации записывается в поле `source`.

Возможные значения:

```text
fallback_category_type_popular
fallback_category_popular
fallback_type_popular
fallback_brand_popular
fallback_global_popular
```

Так мы можем отличить, почему товар попал в итоговый список.

## Как происходит объединение

На вход `FallbackMerger` получает уже ранжированные поведенческие рекомендации.

Дальше для каждого `item_id` он:

1. Берёт уже найденные поведенческие рекомендации.
2. Убирает дубли.
3. Не допускает рекомендацию товара самому себе.
4. Проверяет, хватает ли рекомендаций до `top_k`.
5. Если не хватает, добавляет кандидатов из резервных списков.
6. Пересчитывает `rank` от 1 до `top_k`.

Поведенческие рекомендации сохраняются первыми. Резервные кандидаты добавляются только в свободные места.

## Настройки

Настройки резервного слоя находятся в конфиге:

```yaml
business:
  fallback:
    enabled: true
    top_k: 20
    candidate_pool_size: 200
    global_candidate_pool_size: 200
    metadata_candidate_pool_size: 200
    min_item_events: 1
    enable_category_type: true
    enable_category: true
    enable_type: true
    enable_brand: false
    enable_global: true
    include_cold_start_items: true
    include_catalog_only_sources: false
```

Главные параметры:

| Параметр                       | Что означает                                                                    |
|--------------------------------|---------------------------------------------------------------------------------|
| `enabled`                      | включён ли резервный слой                                                       |
| `top_k`                        | сколько рекомендаций нужно получить для каждого товара                          |
| `candidate_pool_size`          | общий размер пула кандидатов                                                    |
| `global_candidate_pool_size`   | размер глобального пула популярных товаров                                      |
| `metadata_candidate_pool_size` | размер пулов по метаданным                                                      |
| `min_item_events`              | минимальное число событий у товара-кандидата                                    |
| `enable_category_type`         | использовать совпадение категории и типа                                        |
| `enable_category`              | использовать совпадение категории                                               |
| `enable_type`                  | использовать совпадение типа                                                    |
| `enable_brand`                 | использовать совпадение бренда                                                  |
| `enable_global`                | использовать глобально популярные товары                                        |
| `include_cold_start_items`     | добавлять товары из `item_popularity`, даже если у них нет поведенческой выдачи |
| `include_catalog_only_sources` | добавлять товары только из каталога, даже если по ним нет событий               |

Подробнее о настройках: [`../../../configs/README.md`](../../../configs/README.md).

## Холодные товары

Параметр `include_cold_start_items` отвечает за товары, которые есть в статистике популярности, но не получили
поведенческих рекомендаций.

Если он включён, такие товары тоже попадают в итоговый список источников, для которых нужно собрать рекомендации.

Это полезно для товаров с малым количеством событий: поведенческих пар по ним может не быть, но показать похожие или
популярные товары всё равно нужно.

## Товары только из каталога

Параметр `include_catalog_only_sources` отвечает за товары, которые есть в `product_information`, но не встречались в
событиях.

В текущей версии по умолчанию он выключен.

Это осторожное поведение: если по товару нет пользовательских действий, мы не всегда хотим автоматически создавать для
него выдачу. Такой режим можно включать отдельно, когда нужно покрывать весь каталог.

## Что происходит со `score`

Резервные рекомендации получают:

```text
score = 0.0
```

Это сделано намеренно.

Мы не смешиваем поведенческую оценку похожести и резервные бизнес-правила. Если рекомендация пришла из поведения
пользователей, её `score` рассчитан в [`retrieval`](../retrieval/README.md). Если рекомендация добавлена резервным
слоем, её происхождение видно по `source`.

## Пример использования

```python
from ozon_similar_products.business.fallback import (
    FallbackConfig,
    FallbackLayer,
)

fallback_config = FallbackConfig(
    enabled=True,
    top_k=20,
)

fallback_layer = FallbackLayer(config=fallback_config)

final_recommendations = fallback_layer.apply(
    recommendations=behavioral_recommendations,
    item_popularity=item_popularity,
    product_information=product_information,
)
```

Если резервный слой выключен, `FallbackLayer` просто возвращает входные рекомендации после проверки контракта.

## Где находится в конвейере

Резервный слой применяется после выбора top-K поведенческих кандидатов.

```text
retrieval
→ behavioral recommendations
→ business fallback
→ final recommendations
→ output
```

То есть сначала мы строим максимально честную поведенческую выдачу. И только потом добираем недостающие места резервными
правилами.

## Границы ответственности

Что делает `business`:

* строит резервные списки кандидатов;
* использует популярность товаров;
* использует метаданные товаров;
* дополняет короткие списки рекомендаций;
* сохраняет поведенческие рекомендации выше резервных;
* помечает источник резервной рекомендации через `source`.

Что не делает `business`:

* не читает сырые данные;
* не очищает события;
* не строит сессии;
* не создаёт пары товаров;
* не рассчитывает поведенческий `score`;
* не сохраняет итоговые файлы;
* не выдаёт рекомендации наружу.

Эти задачи находятся в других слоях:

| Задача                         | Модуль                                        |
|--------------------------------|-----------------------------------------------|
| чтение данных                  | [`data`](../data/README.md)                   |
| очистка событий и сессии       | [`preprocessing`](../preprocessing/README.md) |
| популярность товаров           | [`features`](../features/README.md)           |
| поведенческие рекомендации     | [`retrieval`](../retrieval/README.md)         |
| сохранение результата          | [`output`](../output/README.md)               |
| получение готовых рекомендаций | [`serving`](../serving/README.md)             |

## Что менять осторожно

| Что менять                     | Почему осторожно                                        |
|--------------------------------|---------------------------------------------------------|
| `enabled`                      | меняет сам факт добавления резервных рекомендаций       |
| `top_k`                        | должен согласовываться с размером итоговой выдачи       |
| `min_item_events`              | влияет на качество и покрытие резервных кандидатов      |
| `enable_global`                | глобально популярные товары могут стать слишком частыми |
| `enable_brand`                 | бренд может быть полезен не для всех категорий          |
| `include_cold_start_items`     | увеличивает покрытие товаров с малым числом событий     |
| `include_catalog_only_sources` | может создать рекомендации для товаров без поведения    |
| порядок уровней fallback       | влияет на то, какие кандидаты попадут раньше            |

После изменения резервных правил нужно смотреть не только общие метрики качества, но и долю рекомендаций с `source`,
начинающимся на `fallback_`.

Подробнее о метриках: [`../../../docs/evaluation_metrics.md`](../../../docs/evaluation_metrics.md).

## Быстрая проверка

```python
from ozon_similar_products.business.fallback import (
    FallbackConfig,
    FallbackLayer,
)

config = FallbackConfig(
    enabled=True,
    top_k=20,
    enable_category_type=True,
    enable_category=True,
    enable_type=True,
    enable_brand=False,
    enable_global=True,
)

layer = FallbackLayer(config=config)

result = layer.apply(
    recommendations=recommendations,
    item_popularity=item_popularity,
    product_information=product_information,
)

print(result.head())
print(result["source"].value_counts())
```

Для полного запуска лучше использовать команды из [`../../../scripts/README.md`](../../../scripts/README.md).

## Коротко

Мы используем `business`, чтобы не оставлять товары без рекомендаций, если поведенческих данных не хватило.

Сначала проект строит рекомендации по поведению пользователей.

Потом резервный слой добирает недостающие места популярными товарами из той же категории, типа, бренда или из общего
списка популярных товаров.

Источник такой рекомендации всегда записывается в `source`, а `score` остаётся равным `0.0`, чтобы не смешивать
поведенческую оценку и резервные правила.
