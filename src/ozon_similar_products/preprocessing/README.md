# Предобработка событий

В этом модуле сырые пользовательские события превращаются в таблицы, с которыми уже можно строить похожие товары.

После слоя [`data`](../data/README.md) у нас есть исходные события из логов. Но в таком виде они ещё не подходят для
расчёта рекомендаций: в них могут быть повторы, нет единого формата времени, встречаются нетоварные действия, а действия
одного пользователя ещё не сгруппированы в сессии.

Модуль `preprocessing` решает две задачи:

1. очищает события и оставляет только товарные действия;
2. собирает пользовательские сессии по времени.

## Что делает модуль

```text
сырые события пользователей
→ очищенные товарные события
→ пользовательские сессии
```

В текущей версии здесь находятся два основных класса:

```text
EventCleaner
SessionBuilder
```

`EventCleaner` готовит события к дальнейшей обработке.

`SessionBuilder` группирует события одного пользователя в короткие временные контексты — сессии.

## Основные файлы

| Файл                                     | Что в нём находится                                              |
|------------------------------------------|------------------------------------------------------------------|
| [`clean_events.py`](clean_events.py)     | очистка сырых событий и приведение их к контракту `clean_events` |
| [`build_sessions.py`](build_sessions.py) | построение пользовательских сессий из очищенных событий          |

## Очистка событий

### `EventCleaner`

Класс `EventCleaner` принимает сырые события и возвращает очищенные товарные события.

На вход подаётся таблица с колонками из контракта сырых событий.

На выходе получается таблица с колонками:

```text
user_id
event_date
timestamp
action_type
item_id
search_query
widget_name
```

Во время очистки мы:

* проверяем, что во входной таблице есть нужные колонки;
* убираем полные дубли строк;
* приводим `timestamp` к единому формату времени;
* убираем события без `user_id`;
* убираем события без времени;
* убираем события без `action_type`;
* оставляем только товарные действия;
* убираем события без `item_id`;
* добавляем `event_date` из времени события;
* заполняем пустой `widget_name` значением `unknown`;
* проверяем, что результат соответствует контракту очищенных событий.

Пример использования:

```python
from ozon_similar_products.preprocessing.clean_events import EventCleaner

cleaner = EventCleaner(
    item_action_types=["view", "click", "favorite", "to_cart"],
)

events_clean = cleaner.transform_day(raw_events)
```

Для нескольких дневных таблиц:

```python
events_clean = cleaner.transform_window([
    raw_events_day_1,
    raw_events_day_2,
])
```

Контракты raw events и clean events описаны в [`../../../docs/data_contract.md`](../../../docs/data_contract.md).

## Какие действия остаются после очистки

В рекомендациях мы используем только действия, связанные с конкретным товаром:

```text
view
click
favorite
to_cart
```

Действие `search` может встречаться в исходных логах, но оно не является товарным действием, потому что не всегда
связано с конкретным `item_id`. Поэтому на этом этапе оно не попадает в очищенные события для построения похожих
товаров.

Список товарных действий задаётся в конфигурации:

```yaml
events:
  item_action_types:
    - view
    - click
    - favorite
    - to_cart
```

Подробнее о настройках: [`../../../configs/README.md`](../../../configs/README.md).

## Построение сессий

### `SessionBuilder`

Класс `SessionBuilder` строит пользовательские сессии из очищенных событий.

Сессия — это последовательность действий одного пользователя, между которыми прошло не больше заданного числа минут.
Если пауза между соседними действиями больше порога, начинается новая сессия.

Пример:

```python
from ozon_similar_products.preprocessing.build_sessions import SessionBuilder

builder = SessionBuilder(timeout_minutes=20)

sessions = builder.transform_day(events_clean)
```

Можно создать объект из конфигурации проекта:

```python
builder = SessionBuilder.from_config(config)
```

Параметр берётся из `pipeline.session_timeout_minutes`.

## Что появляется в таблице сессий

На выходе `SessionBuilder` возвращает таблицу с колонками:

| Поле                 | Что означает              |
|----------------------|---------------------------|
| `user_id`            | пользователь              |
| `session_index`      | номер сессии пользователя |
| `session_start_date` | дата начала сессии        |
| `event_date`         | дата конкретного события  |
| `timestamp`          | время события             |
| `action_type`        | тип действия              |
| `item_id`            | товар                     |

Внутри построения сессий события сортируются по пользователю, времени, товару и типу действия. После этого для каждого
пользователя считается разрыв между соседними действиями. Если разрыв больше заданного порога, начинается новая сессия.

## Почему сессии строятся после очистки

Мы сначала очищаем события, а только потом строим сессии.

Так проще контролировать качество данных:

```text
сырые события
→ убрали нетоварные и неполные строки
→ получили события с item_id
→ построили сессии только по товарным действиям
```

Если строить сессии сразу на сырых логах, в них попадали бы события поиска и другие действия, которые не участвуют в
построении похожих товаров. Это могло бы растягивать сессии и добавлять шум в пары товаров.

## Сессии на границе дней

В модуле есть два режима:

```text
transform_day
transform_window
```

`transform_day` строит сессии для одной таблицы событий.

`transform_window` сначала объединяет несколько дневных таблиц, а потом строит сессии. Это важно для событий около
полуночи: если пользователь сделал одно действие в конце одного дня, а другое в начале следующего, они могут остаться в
одной сессии, если разрыв между ними меньше заданного порога.

В полном конвейере обработки дополнительно используется потоковая логика, чтобы не загружать весь период в память сразу.
Но смысл остаётся тем же: сессия определяется пользователем, порядком событий и временным разрывом между ними.

Подробнее о потоковом запуске: [`../pipeline/README.md`](../pipeline/README.md).

## Границы ответственности

Модуль `preprocessing` не считает похожесть товаров и не выбирает рекомендации.

Он отвечает за подготовку событий:

```text
raw events
→ clean events
→ sessions
```

Что делает `preprocessing`:

* удаляет полные дубли событий;
* нормализует время события;
* оставляет только товарные действия;
* убирает события без пользователя, времени, типа действия или товара;
* добавляет дату события;
* строит сессии по временным разрывам;
* проверяет контракты выходных таблиц.

Что не делает `preprocessing`:

* не считает популярность товаров;
* не строит пары товаров;
* не назначает веса действиям;
* не рассчитывает итоговую оценку похожести;
* не выбирает top-K рекомендаций;
* не добавляет резервные рекомендации.

Эти задачи начинаются в следующих слоях:

| Задача                       | Модуль                                |
|------------------------------|---------------------------------------|
| популярность товаров         | [`features`](../features/README.md)   |
| пары товаров внутри сессий   | [`retrieval`](../retrieval/README.md) |
| агрегация и расчёт похожести | [`retrieval`](../retrieval/README.md) |
| резервные рекомендации       | [`business`](../business/README.md)   |
| сохранение результата        | [`output`](../output/README.md)       |

## Как этот слой используется дальше

Очищенные события используются сразу в нескольких местах.

```text
clean_events
  → sessions
  → item popularity
  → action type distribution
```

Сессии затем передаются в построение пар товаров:

```text
sessions
→ item pairs
→ pair aggregates
→ pair scores
→ recommendations
```

То есть `preprocessing` задаёт основу для всего дальнейшего расчёта похожести.

## Что менять осторожно

| Что менять                      | Почему осторожно                                              |
|---------------------------------|---------------------------------------------------------------|
| список товарных действий        | от него зависит, какие события попадут в рекомендации         |
| правила удаления строк          | можно случайно потерять полезные события или пропустить мусор |
| приведение `timestamp`          | ошибки во времени ломают сессии                               |
| `session_timeout_minutes`       | сильно влияет на длину сессий и количество пар товаров        |
| сортировку событий              | от неё зависит порядок действий внутри сессии                 |
| контракт `CLEAN_EVENTS_COLUMNS` | на него опираются следующие слои                              |
| контракт `SESSIONS_COLUMNS`     | по нему строятся пары товаров                                 |

Если меняется логика сессий, нужно перепроверить модуль [`retrieval`](../retrieval/README.md), потому что именно он
превращает события внутри сессии в пары товаров.

## Быстрая проверка

Пример ручной проверки на небольшом фрагменте данных:

```python
from ozon_similar_products.data import load_events
from ozon_similar_products.preprocessing.clean_events import EventCleaner
from ozon_similar_products.preprocessing.build_sessions import SessionBuilder

raw_events = load_events(sample_days=1, sample_rows=1000)

cleaner = EventCleaner(
    item_action_types=["view", "click", "favorite", "to_cart"],
)
events_clean = cleaner.transform_day(raw_events)

builder = SessionBuilder(timeout_minutes=20)
sessions = builder.transform_day(events_clean)

print(events_clean.head())
print(sessions.head())
```

Для полного запуска лучше использовать команды из [`../../../scripts/README.md`](../../../scripts/README.md).

## Коротко

Мы используем `preprocessing`, чтобы превратить сырые события в надёжную основу для рекомендаций.

Сначала `EventCleaner` оставляет только корректные товарные действия.

Потом `SessionBuilder` группирует эти действия в пользовательские сессии.

После этого модуль [`retrieval`](../retrieval/README.md) уже может строить пары товаров внутри сессий.
